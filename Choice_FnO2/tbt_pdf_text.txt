TBT (Tick-By-Tick) WebSocket Feed - Integration
Guide
Audience: Subscriber / consumer teams integrating with the TBT WebSocket feed Status: Draft (test
environment) Last updated: 2026-06-12
1. Abstract
This document describes the TBT (Tick-By-Tick) WebSocket feed strictly from a subscriber's point of view:
how to connect, what to send, and what you get back. It does not cover server internals.
It builds on the earlier "Iris Documentation" PDF and the standalone client.js shared with the web team,
and covers the current tbt-test URL, the subscribe / unsubscribe / pause / resume messages, the
optional requestId field, subscription and mode behaviour, limits, and known edge cases.
2. Overview
A single WebSocket connection gives you streaming market data:
• You send small JSON control messages (subscribe / unsubscribe / pause / resume).
• You receive market-data updates as binary frames, one frame per tick per instrument.
The server transmits binary, not JSON. The previously shared client.js is the reference client: it opens
the connection, sends the subscribe message, and decodes each binary frame into a plain object (the
JSON shapes shown in Section 10). You can use client.js as-is, or port its parser to your language
using the byte layouts in Section 10.
3. Prerequisites
• A WebSocket client capable of:
◦ Sending text frames (your JSON control messages).
◦ Receiving binary frames (market data).
◦ Replying to WebSocket ping control frames with pong (standard libraries such as ws for
Node, the browser WebSocket, Go gorilla/websocket, and Python websockets do
this automatically).
• A clientId issued to you (for example X04455).
• A JWT token (see Section 5; currently not enforced on test, but build for it).
• Ability to parse little-endian fixed-width binary (int32, int64, uint32, float64, fixed-length byte
arrays) if you write your own decoder.
• Network connectivity to the host machine. The feed is deployed on an AWS machine. Your
client must be able to reach that machine on port 8085. For current testing we run client.js
from a server that has outbound access to port 8085 on the AWS host where the feed is deployed.
Make sure your environment (firewall / security group / VPN) allows that path before integrating.

5. Authentication (currently disabled on test)
Authentication is switched off in the test environment right now, so you can connect without a valid token.
Plan for it to be turned back on. When enabled, the JWT is validated during the connection handshake,
before the WebSocket upgrade. Failures return an HTTP 401 with a message such as:
• clientId is required
• token is required
• invalid token (bad signature or unparseable)
• invalid issuer (issuer is not https://sso.choiceindia.com)
• expired token (the exp claim is in the past)
Edge case: auth failures occur at the HTTP handshake, so you get an HTTP 401 response, not a
WebSocket close. Handle a failed handshake separately from a post-connect disconnect.

7.1 Subscribe
{ "action": "subscribe", "mode": "full", "instruments": ["2@62607"] }
On acceptance, if a recent value is available you immediately receive a snapshot frame for each newly-
subscribed instrument (so you get current state without waiting for the next tick), then live updates follow.
7.2 Unsubscribe
{ "action": "unsubscribe", "mode": "full", "instruments": ["2@62607"] }
mode must match the mode the instrument is currently subscribed under. Unsubscribing something you
are not subscribed to is a harmless no-op.
7.3 Pause / Resume
{ "action": "pause" }
{ "action": "resume" }
• pause stops market-data frames for this connection. The connection stays alive (pings continue)
and your subscriptions are retained.
• resume restarts market-data frames for your existing subscriptions.
• mode and instruments are not required and are ignored.
• Pausing while already paused (or resuming while not paused) is a harmless no-op.
Pause edge case: frames produced while you are paused are dropped, not buffered. On resume you
receive fresh ticks going forward, not a backlog.
8. Instrument identifier format
Each instrument is a string: <SegmentID>@<ExchangeToken>
• Validated against the pattern ^\d+@\d+$ (digits, one @, digits). Anything else is rejected (Section
11).
• SegmentID is the numeric segment code (table below).
• ExchangeToken is the exchange's numeric token. For indices, use the index token as provided by
the exchange (for example 1@26000 for NIFTY 50).
Segment IDs

Examples: 1@26000 (NSE Cash, token 26000), 2@62607 (NSE F&O), 5@562396 (MCX F&O).
9. Modes
Mode Depth Binary frame size
touchline Best bid/ask + last trade price 94 bytes
best_five 5 bid + 5 ask levels 214 bytes
full 20 bid + 20 ask levels + full stats / OI 631 bytes
You identify the mode of an incoming frame by its byte length (94 / 214 / 631). There is no type header. Any
other length is unexpected and should be logged and skipped.
10. What you receive (market-data frames)
Frames arrive as binary, little-endian, fixed-size structures with no padding between fields. The client
decodes each frame into an object. For each mode below you get: the decoded object (what your code
sees after parsing) and the exact byte layout (what you parse).
Common conventions
• Endianness: little-endian for all multi-byte numbers.
• Prices are raw integers. Divide by price_divisor (a uint32 present in every frame) to get the
real price. Example: raw 2398000, divisor 100, gives 23980.00.
• token is a fixed 50-byte UTF-8 field, null-padded. Trim trailing \0 to get the string.
• segment is the numeric SegmentID (Section 8).
• A level (one price row) is 12 bytes: int32 price, int32 quantity, int32 total_orders.
• Timestamps: last_update_timestamp and last_trade_timestamp are nanoseconds since
epoch (int64). timestamp (full mode only) is the server send time in milliseconds (int64).

10.4 Reference decoder
The previously shared client.js (Node, ws) is the canonical parser and matches the layouts above. It
switches on buf.length (94 / 214 / 631) to pick the parser, reads each field with readInt32LE /
readBigInt64LE / readDoubleLE / readUInt32LE, and treats any non-binary frame as a text status/error
message. Use it as a port target for other languages.
11. Validation and silent rejections

message back. Do not wait for an error frame. If you sent a malformed request, you simply will not get data.

12. Subscription behaviour and mode strength
Modes have a strength order: touchline (weakest) < best_five < full (strongest). The feed keeps one
effective subscription per instrument, at the strongest mode you requested.
• Upgrade: if 2@62607 is subscribed as touchline and you then subscribe it as full, it moves to
full. You then receive only full frames for it.
• Downgrade ignored: if it is on full and you subscribe it as touchline, it stays on full (the
weaker request is treated as already covered).
• Same-mode duplicate: re-subscribing in the mode it is already in is ignored.
• Counting toward the 1000 limit: an instrument uses exactly one slot, regardless of how many
times or modes you sent it.
Practical guidance: pick the single mode you want per instrument. To change a token's mode, just
subscribe it in the new mode; there is no need to unsubscribe first (though doing so is harmless).
13. Heartbeat / keep-alive
The connection is kept alive with WebSocket ping/pong control frames (not your JSON messages):
Parameter Value Meaning
Server ping period ~27 s The server sends a ping roughly every
27 seconds.
Pong wait 30 s You must respond with a pong
(automatic in standard libraries). If no
pong is seen within 30 s of the last
one, the server closes the connection.
What you need to do:
• Use a library that auto-responds to ping with pong (the default for the libraries listed in Section 3).
If you handle frames manually, you must reply to pings.
• Optionally, if you receive no ping for about 30 s, treat the connection as stale and reconnect.
• Pings continue even while you are paused.
14. Reconnection behaviour
• No session resume. If the connection drops and you reconnect, you start clean. You must re-send
all your subscribe messages. A disconnected client's subscriptions are not remembered.
• Snapshot on (re)subscribe. Each fresh subscribe gives you the latest available frame immediately
(if any), so you recover current state quickly.
• Keep your read loop fast. Each connection has a bounded outbound buffer. If you read too slowly
and it fills, new frames are dropped for your connection rather than blocking or disconnecting you.
You can silently miss ticks if you cannot keep up, so offload heavy processing off the read path.
• Close frames. A clean close uses a normal WebSocket close frame; abnormal closures (network
or heartbeat timeout) may arrive without a reason.
15. Quick start checklist
1. Confirm network connectivity to the AWS host on port 8085 (Section 3).
2. Open wss://tbt-test.choiceindia.com:8085/v1/websocket?clientId=<ID> (add
&token=<JWT> when auth is enabled).
3. For TLS on test, allow the non-public certificate (test only).
4. On open, send a subscribe text frame, for example
{"action":"subscribe","mode":"touchline","instruments":["1@26000"]}.
5. On each binary message, branch on length (94 / 214 / 631) and parse per Section 10 (little-endian).
6. Convert prices using price_divisor; trim the 50-byte token.
7. Make sure your library auto-replies to pings.
8. Handle text frames as status/error (for example the subscription-limit message).
9. On disconnect, reconnect and re-subscribe everything.
